# Wiki 拆分指导（政策法规版）

本仓库针对医疗器械、药品监管等政策法规类文档。流程借鉴 [ccic-llm-wiki](./ccic-llm-wiki/wiki-拆分指导.md)，并按「章 / 节 / 条」与跨规范引用做了适配。

**给 Agent**：用户要求「按 `wiki-拆分指导.md` 执行」时，即完成下文 **§7 完整工作流**（脚本拆分 → 补摘要 → 补关联）。只读写在 `wiki/`；禁止为查询或补全去读 `raw/`、`data/`、`clean/` 等。处理全库时按规范**连续执行**各步，完成后统一汇报，勿每部规范都停下来询问是否继续。

**索引原则**：`-index.md` 用于快速定位；条目录只列条号与页码；**条文摘要**写清「本条讲什么」（**约 50 字**）；正文保留完整条款。

---

## 1. 目录组织方式

```text
wiki/
└── 药物警戒质量管理规范/
    ├── 第一章总则/
    │   ├── 总则.md
    │   └── 总则-index.md
    ├── 第四章监测与报告/
    │   ├── 第一节报告与处置.md
    │   ├── 第一节报告与处置-2.md    # 该节超过 2 页才分页
    │   └── …
    └── 药物警戒质量管理规范-index.md
└── 医疗器械委托生产质量协议编制指南/
    ├── 2.范围/
    │   ├── 2.1….md
    │   └── …
```

说明（对齐 ccic-llm-wiki）：

- **`#` 一级标题** → 章节目录；**`##` 二级标题** → 章内正文文件（一节一文件）。
- **章内无 `##`** → 该章一个正文文件（如 `总则.md`）。
- **超过 2 页** 的章/节才继续 `-1`、`-2` …；相邻段 **1 页重叠**，续段补回标题行。
- **无任何章级 `#`**：整篇扁平按页拆分（`<规范名>-1.md` …）。
- 每个正文文件有 `-index.md`；规范根有 `<规范名>-index.md`。

## 2. 脚本拆分

```bash
python scripts/split_wiki.py raw/<规范名>.md
python scripts/batch_split_wiki.py --report
```

- 默认 **2 页/节（或章）**；`--max-pages-per-section` 可改（ccic 建筑规范默认 5）。
- 页码：`<!-- 第 N 页 -->` 由 `ocr-plan` 插在**每页末尾**（非页首）。**标记之后**的正文属于第 **N+1** 页；首个标记之前为第 1 页。可与正文同行（如 `<!-- 第 1 页 -->量管理…` 表示换页后续文）。`split_wiki.py` 的条目录页码按此语义计算；页数按段内**去重页码**统计。
- **已建好 wiki、不想重拆**：运行 `python scripts/fix_wiki_page_numbers.py`（可加 `--dry-run` 预览），只改各 `-index.md` 中的页码，保留正文与摘要文字。
- 脚本只生成正文与 **条目录 + `TODO: 待LLM补写` 占位**；语义摘要与关联由 **LLM 按本指导 §4～§5** 补全。

### 2.1 标题与拆分单元

| 层级 | 参与拆分 | 进入条目录 |
| --- | --- | --- |
| 章 `# 第X章` / `# 2.范围` | 一级目录 | 否 |
| 节 `## 第X节` / `## X.X` | 正文文件名 | 否 |
| 条 `第X条` | 否 | **是**（仅条号+页码） |
| 指南 `4.1` / `4.1.1` | 否 | **是**（编号+页码） |

### 2.2 源文件纠错

见 `split_wiki.py` 中 `normalize_title` / `normalize_inline_ocr`；正文前目录、尾部用词说明不并入最后一章。

### 2.3 纯表格类文档（如「检查要点」）

- 几乎全是 `<table>` 时 → `-表1`、`-表2` …；表间页码/「表格（第 N 页）」**不单独成文件**。
- 索引表名优先「表格第 N 页」。

### 2.4 中文序号（「一、」「（一）」）

- `## 一、…` 或行首 `二、…` → **章**；`（一）…` → **节**（`第X章` 下的 `（一）` 为法条列举，不单独成节）。
- 文首 `# 规范全名` 为总标题，不当章。

### 2.5 过小二级节合并

- 非空行 &lt; 8 或字符 &lt; 350 → 与相邻小节合并为 `TODO-待LLM命名-N.md`，由 LLM 重命名为 8～24 字简称。
- 阈值见 `split_wiki.py`：`MIN_SECTION_BODY_LINES` / `MIN_SECTION_BODY_CHARS`。

### 2.6 HTML 表格不得从中间切开

- `<table>…</table>` 整表 → `<节名>-表N.md`；表间正文仍可按页拆。

### 2.7 无「第X章」时

| 情形 | 处理 |
| --- | --- |
| `# 第一章` / 附则 / 附录 | 一级目录 = 章 |
| `# 2.范围` 等数字篇 | 一级目录 = 篇 |
| 全文无章级 `#` | 扁平 `<规范名>-1.md` … |
| 章内第一个 `##` 前有附件/表 | 章前导段（含 `-表N`） |

**建议在 `raw/` 校验**：补章标题、理顺 OCR 乱序章节、短通知可保留扁平并在根索引说明。

### 2.8 脚本拆分后检查

```bash
python scripts/count_split_lines.py raw/<规范名>.md --details
```

- 单节多页却无 `-1/-2`、有大表却无 `-表N` → 重跑拆分或查 `raw` 标题。
- 半表：各文件 `<table` 与 `</table>` 成对。

## 3. index 文件结构

### 3.1 正文 `-index.md`

```markdown
# 第一章总则-1索引

## 条目录

- 第一章总则
  - 第一条（第1页）
  - 第二条（第1页）

## 条文摘要

- **第一条**（第1页）：<约 50 字，1～2 句：适用对象、义务类型、核心要求…>
```

- **条目录**：只写 `条号（第N页）`，不写条文原文截断。
- **条文摘要**：供检索判断「本条讲什么」；不是一句话标题。

合并块 `-index.md` 另含 **合并来源** 列表及 `TODO: 待LLM命名——…`（见 §4.1）。

### 3.2 根 `<规范名>-index.md`

```markdown
## 章节目录        # 办法类：到节
## 条目录          # 指南类：全篇条号
## 文件目录        # 各正文文件及页码
## 各文件摘要      # 每文件约 50 字
```

### 3.3 正文末尾（LLM 补，见 §5）

```markdown
## 关联文件

- [第二节….md](../第三章…/第二节….md)
```

## 4. LLM 补全：索引与摘要

**建议先于关联完成**（各文件摘要写好后，关联质量更高）。用户若只要求「拆分+摘要」，可只做本节。

### 4.1 合并块命名（`TODO-待LLM命名-*.md`）

1. 读 **合并来源** 与正文；拟定 **8～24 字** 文件名。
2. 重命名 `.md` / `-index.md`；更新根 **文件目录**。
3. 将 `TODO: 待LLM命名——…` 改为正常 **条文摘要** / **各文件摘要**。

### 4.2 条文摘要

每条尽量包含：**主题**、**行为类型**（义务/禁止/责任等）、**核心要求**（可省略与上下条关系以控字数）。

- **约 50 字**，**1～2 句**；格式 `- **第X条**（第N页）：…`（页码保留在条目前，摘要正文里不要重复写页码）
- **禁止**：合并多条、照抄首句、空泛「规定了相关要求」、超过 70 字仍不截断。
- 具体罚款、表格、精确数字留在正文。

### 4.3 各文件摘要

该拆分段/节整体主题、义务类型或典型查询场景（**约 50 字**，1～2 句）。

### 4.4 执行顺序（单部规范）

```text
① TODO-待LLM命名-* → 重命名 + 改根文件目录
② 每个正文 *.md + 配对 *-index.md → 条目录、条文摘要
③ <规范名>-index.md → 各文件摘要
④ rg "TODO: 待LLM补写" wiki/<规范名>/ 应为 0
```

**特殊体例**：`-表N.md` 写表名、用途、栏位；数字篇指南按 `4.1` 等编号；扁平拆分写清页段主题。

## 5. LLM 补全：关联文件

**前置**：该规范 §4 已完成（`各文件摘要` 已齐）。

### 5.1 格式

- 标题固定 `## 关联文件`（勿写「可选」）。
- `[文件名.md](相对路径)`；路径须存在；通常 **3～6 条**。

### 5.2 如何判定关联

1. 读当前正文、`-index.md`、根 **各文件摘要**；必要时看他规范根索引。
2. **硬引用**：「第 X 条」「《XX 办法》」、附表号 → 必链。
3. **语义**：只读当前节答不全（缺定义、时限、责任、跨规范依据）再链。
4. **不链**：主题仅相近、当前节已能答全。

| 类型 | 示例 |
| --- | --- |
| 同规范上下游 | 义务 ↔ 法律责任 |
| 同规范相邻 | `申办者-1` ↔ `申办者-2` |
| 跨规范 | 指南 / 检查要点 → 办法 |
| 表格 | `-表N` ↔ 填报说明 |

查询时：index → 正文 → **不够再读**关联（渐进式披露，见 `wiki-查询指导.md`）。

### 5.3 执行顺序

```text
① 确认无 TODO: 待LLM补写
② 每个正文 *.md 文末写/更新 ## 关联文件
③ 无跨节引用时可写：- （本节无额外关联）
```

## 6. 全部完成后检查

```bash
rg "TODO: 待LLM" wiki/<规范名>/
rg "TODO-待LLM命名" wiki/<规范名>/
python scripts/count_split_lines.py raw/<规范名>.md --details
```

- 条目录无长段原文；摘要能支撑不打开正文时的相关性判断。
- 各正文有 `## 关联文件`（或注明无额外关联）；链接可解析。
- 单文件跨多页却未拆 → 重跑 `split_wiki.py`。

## 7. Agent 完整工作流（llm-wiki 导入）

用户说「按拆分指导处理 wiki」或「导入/补全 wiki」时，按下列顺序执行（可指定一部或全库）：

| 步骤 | 操作 | 产出 |
| --- | --- | --- |
| 0 | 确认 `raw/<规范名>.md` 已人工校验 | — |
| 1 | `python scripts/split_wiki.py raw/<规范名>.md` 或 `batch_split_wiki.py --report` | 正文 + 占位 index |
| 2 | `count_split_lines.py` 抽查 | §2.8 |
| 3 | LLM：**§4** 索引与摘要（含合并块命名） | 无 `TODO: 待LLM补写` |
| 4 | LLM：**§5** 关联文件 | 文末 `## 关联文件` |
| 5 | **§6** 自检 | — |
| 6 | 可选：`wiki-查询指导.md` spot-check | — |

**全库**：对 `wiki/` 下每部规范重复步骤 1～5（已完整且无 TODO 的可跳过）。**一部规范内先 §4 再 §5**，逻辑上分步、执行上可一次对话做完。

**汇报**：每规范一行——规范名、正文文件数、是否仍有 TODO、关联是否已补。

## 8. Claude Code 提示词（可复制）

**前提**：`raw/` 已人工校验；`batch_split_wiki.py` 或 `split_wiki.py` 已跑完（当前脚本页码与 OCR 一致：`<!-- 第 N 页 -->` 在页末，其后为第 N+1 页）。规则以本文为准，提示词只定范围。

### 8.0 全库先拆分（终端，可选）

```bash

```

② 再补各 *-index.md：保留条目录页码不动，将「条文摘要」中 TODO: 待LLM补写 改为约 50 字摘要（1～2 句：主题、义务类型、核心要求）
③ 最后补 <规范名>-index.md 的「各文件摘要」
约束：只读写 wiki/<规范名>/；禁止读 raw/ 做查询式阅读。

### 8.2 单部规范 · 补关联（§5）

```text

```

### 8.3 单部规范 · 一条龙（拆分已做完时用）

```text
按 wiki-拆分指导.md §7，只完成 wiki/<规范名>/ 的 LLM 部分（假定已 batch/split 拆分）：
§4 摘要（含 TODO-待LLM命名）→ §5 关联 → §6 自检（rg TODO、count_split_lines）。
只读写该规范目录；禁止读 raw/。
单次 LLM 编辑不超过 20 个文件，超出则汇报进度等我「继续」。
最终 8 行内汇报：正文文件数、TODO 是否清零、关联是否已补。
```

### 8.4 全库补全（分批，防 token 超限）

```text
按 wiki-拆分指导.md §7 维护全库 LLM 补全（假定 batch_split_wiki 已跑完）：
1. 列出 wiki/ 下各部规范，标出仍有 TODO: 待LLM补写 或 TODO-待LLM命名 的
2. 本次只完整做完列表中第 1 部：<规范名>（§4→§5→§6）
3. 不要在本轮做第二部；结束时给出下一部规范名
我后续发「继续全库，下一部：<规范名>」。
```

### 8.5 续跑

```text
继续 wiki-拆分指导.md：规范名 <规范名>，阶段 <§4 摘要 | §5 关联 | 全库下一部>。
从上次「剩余列表」第一项做起。
```

## 9. 与查询的关系

- **导入/维护**：本指导 + `AGENTS.md` / `CLAUDE.md`。
- **问答**：只读 `wiki/`，见 [`wiki-查询指导.md`](./wiki-查询指导.md) **§6**（先原文摘抄溯源、后结论；`qa-test/question-template.md`）。
- **评测**：`qa-test/evaluate.md`、`qa-test/check-answer-against-wiki.md`。
