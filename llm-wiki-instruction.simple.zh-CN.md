# LLM Wiki（政策法规版本）

一种使用 LLM 构建持久化、可查询知识库的模式。本文件是高层思路说明，复制粘贴给你的 LLM Agent（Claude Code、Codex、Cursor 等）即可使用。

## 核心理念

大多数 LLM 与文档结合的体验类似 RAG：上传文档、查询时检索片段、生成答案。这种方式可以工作，但每次回答都是从头重新发现知识，知识不会累积。

这里的思路不同：让 LLM **增量地构建并维护一个持久化的 wiki**。这个 wiki 是一组结构化、互相关联的 markdown 文件，位于你与原始资料之间。当你加入一个新规范时，LLM 不只是为以后检索它建立索引，而是会真正去阅读它，把关键定义、条款、表格、附录写进 wiki，并维护交叉引用。

知识只需要被整理一次，然后持续保持最新状态，而不是每次查询都重新推导。

> 关键区别：**wiki 是一个持久存在、会不断积累的产物。**

## 架构

系统有三层：

**原始资料（Raw sources）**：政策法规的 PDF / Word 经过 `ocr-plan` 抽取、`clean-plan` 自动清洗、再由人工校验后，最终放进 `raw/` 目录。`raw/` 之外（`data/`、`raw-0/`、`clean/`）只是流水线产物，**不参与查询**。

**wiki**：由 LLM 生成的 markdown 文件目录，按"规范 → 章 → 节"组织。所有正文文件配套一个 `-index.md` 摘要。这一层完全由 LLM 维护。wiki 页面应尽可能完整地保留原文的条款、定义、时限、法律依据，**不要为了简短而丢信息**。

**schema**：`CLAUDE.md` / `AGENTS.md`、`wiki-拆分指导.md`、`wiki-查询指导.md`。它们告诉 LLM wiki 的结构、约定、工作流。正是它们把 LLM 变成一个有纪律的 wiki 维护者，而不是普通聊天机器人。

## 操作流程

**导入（Ingest）**：把校验过的 `raw/<规范名>.md` 交给 LLM，它会：

1. 先用 `scripts/split_wiki.py` 完成确定性的章节拆分（`#` 章目录、`##` 一节一文件；默认 **2 页/节** 内不拆，超长再 `-1`、`-2` …），生成正文与占位 `-index.md`。
2. 按 [`wiki-拆分指导.md`](./wiki-拆分指导.md) 由 LLM 补全条目录、条文摘要、各文件摘要、合并块命名与 `## 关联文件`（先摘要、后关联，见该文档 §4～§7）。

政策法规文件里的「条款（第X条）」「定义」「时限」「法律依据」是核心信息，应保留在正文文件中，不要摘要稀释。

**查询（Query）**：你针对 wiki 提问。LLM 先读各级 `index.md` 定位相关文件，再深入读取正文文件，最后基于这些 wiki 页面回答。**不再回查 `raw/` 或更上游的资料**。原因是政策法规文件经常几十上百页，查询时再回原文容易把上下文撑爆，也违背了"知识只整理一次"的本意。

**体检（Lint）**：定期让 LLM 给 wiki 做健康检查。检查内容包括：页面重复、失效链接、缺出处、关键条款被多次提到却没有独立页面、相关页面缺交叉引用、原文条款在拆分中被遗漏等。

## 索引

`-index.md` 文件面向内容导航：

- **规范根索引** `wiki/<规范名>/<规范名>-index.md`：列出该规范的章节目录（精确到二级标题）+ 整体摘要。
- **章节索引** `wiki/<规范名>/<章名>/<节名>-index.md`：章节目录精确到 **条**；`## 各条例摘要` 对每条各一行（非整段合并）。

LLM 在回答查询前，先读根索引，再读相关章节索引，最后再决定要不要打开正文。中等规模下，这种 markdown 索引方案足以替代向量检索基础设施。

## 可选项：CLI 工具

到某阶段，你可能想给 LLM 增加一些小工具，例如：

- 全文关键词搜索
- 跨规范引用检查
- 自动化「条款引用 → 目标条款」跳转

当前版本先坚持简单方案：以文件系统和 Markdown 页面为主，不依赖 Obsidian、不依赖图数据库、不依赖向量数据库。

## 技巧与经验

- 原始资料和 wiki 页面分开存放，避免事实源与整理结果混淆。
- 文件命名稳定、清晰，便于后续链接和维护。
- 复杂表格、长定义、关键条款适合单独成页，不要全部塞进同一篇总结。
- 若原文很长，不要试图用一页摘要覆盖全部内容；按章节拆分，每个文件保留可查询的完整正文。
- 有 `##` 时一节一文件；无章内 `##` 时一章一文件；超过 **2 页** 时继续按页拆为 `<名称>-1`、`-2` …（可用 `--max-pages-per-section` 调整）。
- wiki 本质上是 markdown + 文件系统 + Git。天然就有版本历史、分支、协作能力。

## 为什么这样有效

维护知识库最让人厌烦的部分是记账性质的维护工作：更新摘要、维持交叉引用、防止重复。LLM 不会厌倦，可以一次性更新多个文件。wiki 之所以能保持被维护，就是因为维护成本被大幅降低了。

人的工作是筛选资料、引导分析、提出好问题；LLM 的工作则是其他所有整理性的工作。

## 这个项目里的边界

- 当前规模下不依赖向量数据库或图数据库。
- 查询时不回查 `raw/`，只基于 `wiki/`。
- 查询的答案默认不自动写回 wiki（除非用户显式要求"把这条新结论补到 wiki 里"）。
- 适配政策法规类文档，章节识别按 `第X章 / 第X节 / X.X` 处理，详见 `wiki-拆分指导.md`。
